# 独立审计问题识别工程

## Summary
- 新工程独立建仓，仓库名默认定为 `bank-complaint-audit-issues`，本地目录为 `C:\Users\Administrator\Desktop\111\bank-complaint-audit-issues`，与现有清洗工程完全分离。
- 工程目标不是复刻现有 `13/80` 标签，而是自动把清洗后的工单拆成 `issue`，自动发现问题簇，生成可下钻到工单的审计问题类别，并支持后续跨年份、跨地区复用。
- 第一版按“双模式并列”实现：`CPU 基线模式` 保证无 GPU 也能跑；`GPU 增强模式` 在更强机器上提升语义表示、聚类质量和运行速度。
- 第一版不做 Web UI，只交付可独立运行的 Python 工程、CLI、Parquet/CSV/Excel/Markdown 报表、完整 README、requirements 和架构文档；这是最符合生产环境且最利于后续迁移和复用的形态。

## 运行模式与机器要求
- `CPU 基线模式`：
  - 目标机器：`8 核 CPU + 16GB RAM + SSD` 起步，推荐 `8 核以上 + 32GB RAM`。
  - 技术组合：`TF-IDF + jieba + RapidFuzz + 小型中文句向量 + UMAP + HDBSCAN`。
  - 默认模型：`moka-ai/m3e-small`；若机器仍偏弱，则允许仅用 `TF-IDF + 规则 + 聚类` 跑通发现链路。
- `GPU 增强模式`：
  - 目标机器：`12-16 核 CPU + 32-64GB RAM + NVIDIA 12GB VRAM` 起步，推荐 `24GB VRAM`。
  - 技术组合：`BAAI/bge-m3 + TF-IDF + UMAP + HDBSCAN + 稠密/稀疏混合相似度`。
  - 默认模型：`BAAI/bge-m3`，使用 `torch` 本地推理，支持 fp16。
- 当前这台机器只用于写代码、跑 smoke test，不作为全量运行目标；实现时不要求在当前机上完成全量数据聚类。

## 使用的计算机技术
- `Python 3.12`：主语言。
- `pandas + pyarrow`：CSV/Parquet 数据处理与列式落盘。
- `Pydantic`：配置、数据契约、运行参数校验。
- `Typer + Rich`：CLI 和终端输出。
- `DuckDB`：本地分析与报表查询。
- `jieba`：中文分词。
- `PMI + 左右熵 + n-gram 统计`：自动术语发现，不要求先人工做完整词典。
- `RapidFuzz`：术语归一与近似写法合并。
- `regex`：金额、合同、费用、活动、渠道、卡种等强模式抽取。
- `scikit-learn`：`TF-IDF`、相似度计算、基础评估工具。
- `sentence-transformers`：本地开源中文句向量编码。
- `UMAP`：降维。
- `HDBSCAN`：未知类数、长尾场景下的问题发现聚类。
- `c-TF-IDF / 关键词抽取`：聚类命名和代表关键词生成。
- `openpyxl`：导出 Excel 报表。
- “本地开源技术栈”的含义：以上库和模型全部在本地机器运行，不依赖 OpenAI API 或其他外部付费模型服务。

## 核心处理链路
1. `ingest`
- 从 `configs/datasets/jl_2024.yaml` 读取输入路径或 GitHub raw URL。
- 校验必须字段：`ticket_id`、`desc_clean`、`resolution_cause_clean`、`biz_type`、`biz_subtype`。
- 写入 `data/runs/{run_id}/tickets.parquet`，同时落地 `run_manifest.json`。

2. `issue_split`
- 以 `desc_clean` 为主，`resolution_cause_clean` 为辅。
- 使用“中文分句 + 诉求词/争议词/转折词切分 + 邻接回并”生成 `IssueRecord`。
- 每个 issue 必须保留 `ticket_id`、`issue_id`、`issue_text`、`evidence_text`、`source_field`。

3. `term_mining`
- 自动挖掘卡种、系统、活动、费用、合同、机构、合作方等高频术语。
- 输出 `term_catalog.parquet`，包含 `term_text`、`term_type`、`normalized_term`、`support_count`。

4. `extraction`
- 规则和术语联合抽取问题属性，输出 `IssueFacts`。
- 属性字段采用宽表 + JSON 扩展列并存：核心标准字段固定，长尾属性写入 `extra_facts_json`。
- 这里不强制“4 段类码”，而是抽出后续生成展示名所需的事实。

5. `representation`
- 生成两套特征：
  - 稀疏特征：`TF-IDF char 2-4gram + word 1-2gram`
  - 稠密特征：CPU 用 `m3e-small`，GPU 用 `bge-m3`
- 保存到 `feature_store/`，便于后续重复运行不重复编码。

6. `discovery`
- 用 `UMAP + HDBSCAN` 做问题发现。
- 未聚入主簇的 issue 进入 `outlier_handler`，优先按最近簇归并，否则保留为单点工作类，确保全覆盖。
- 每个工作类自动生成：
  - `working_type_id`
  - `auto_title`
  - `keywords`
  - `representative_issue_ids`
  - `cluster_size`

7. `taxonomy`
- 第一版同时维护两层类：
  - `WorkingType`：本次自动发现结果
  - `CanonicalType`：沉淀后的正式问题类型
- 初始允许 `CanonicalType` 为空；系统先跑出吉林 2024 的第一版工作类。
- 后续通过 `promoter.py` 把人工在簇级别确认过的工作类提升为正式类，不做逐条工单标注。

8. `assignment`
- 每个 issue 至少命中一个类，优先级：
  - `CanonicalType` 命中
  - 否则归入本次 `WorkingType`
  - 再不行归入 `singleton working type`
- 同一 issue 允许多标签分配，但第一版限定“一个主类 + 最多两个辅类”，避免结果泛滥。
- 展示类名由 `label_builder.py` 动态拼接：只用实际抽出的属性，不要求固定段数。

9. `reporting`
- 输出：
  - `issues.parquet`
  - `issue_facts.parquet`
  - `working_types.parquet`
  - `issue_assignments.parquet`
  - `category_summary.xlsx`
  - `category_drilldown.xlsx`
  - `new_type_candidates.xlsx`
  - `run_summary.md`
- `category_drilldown.xlsx` 必须包含 `display_name -> ticket_id -> issue_id -> evidence_text`，保证审计可追溯。

## CLI 与对外接口
- 统一 CLI：`audit-issues`
- 命令固定为：
  - `audit-issues ingest --dataset jl_2024 --profile cpu`
  - `audit-issues discover --dataset jl_2024 --profile cpu`
  - `audit-issues assign --dataset jl_2024 --profile cpu`
  - `audit-issues report --dataset jl_2024 --profile cpu`
  - `audit-issues full-run --dataset jl_2024 --profile cpu`
  - `audit-issues promote --dataset jl_2024 --mapping-file reviewed_working_types.csv`
- `reviewed_working_types.csv` 字段固定为：
  - `working_type_id`
  - `action`
  - `canonical_name`
  - `merge_target`
  - `notes`
